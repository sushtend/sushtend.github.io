Tokenomics, data centers, and the economics of future AI infrastructure

From token throughput to grid-scale power: how language-model unit economics and physical infrastructure constrain strategy — not just model capability.

Sushrut Tendulkar · 10 Apr 2026 · Long read

Every word you read from a large language model begins as a token. Tokens are not words — they are sub-word fragments that models process one forward pass at a time, making them a direct proxy for GPU compute, electricity, and ultimately, cost. As providers like OpenAI and Google now process trillions of tokens per day, understanding the physics of tokens is no longer a concern limited to AI engineers. It is a strategic imperative for every executive building on, or competing against, AI-native companies.

The anatomy of a token

To grasp the economics of AI at scale, one must start with the atom of the system: the token. A token is the basic unit that large language models read and write. Contrary to popular assumption, tokens are not equivalent to words. They represent sub-word pieces — fragments of words, individual characters, punctuation, and spaces that a tokenizer carves language into before a model ever sees it.

In English, a widely used rule of thumb is approximately one token per four characters, or about 0.75 words per token. A typical paragraph might contain 80 words but 105 to 110 tokens. This distinction is not pedantic; it has direct financial implications, since cloud providers bill by the token, not the word.

Tokens are the currency of AI. Understanding their physics is now a boardroom issue.

The reason tokens map so directly to cost is architectural. For an autoregressive model — the architecture underpinning today’s dominant LLMs — each new output token requires a full forward pass through the model: matrix multiplications across most model weights, attention computation, and repeated data movement in and out of GPU memory. A useful engineering shorthand is that a decoder-only transformer’s forward pass cost scales roughly linearly with the number of non-embedding parameters, summarised as approximately 2N multiply-add operations per token, where N is the parameter count. This is why a 70-billion-parameter dense model costs far more per token than an 8-billion-parameter model, and why “tokens per second” functions as the standard proxy for GPU expenditure.

Training tokens vs. inference tokens

Not all tokens are created equal in cost structure. Training tokens — used while updating model weights — require a forward pass, a backward pass, and optimizer steps, making them the most compute-intensive. But training is a one-off or periodic expenditure. Inference tokens, consumed every time a user interacts with a deployed model, involve only forward passes. The strategic distinction is critical: training is a capital investment; inference is a running cost that scales directly with user volume.

Within inference itself, two phases govern cost and latency profiles:

  • Prefill: The input prompt is processed to build a key-value (KV) cache. This phase is compute-heavy and relatively fast.
  • Decode: Output tokens are generated one by one using that cache. This phase is memory-bandwidth-bound and highly sensitive to batching strategies and cache management.

This prefill-decode split explains why modern serving systems employ techniques like continuous batching, cache paging (most notably vLLM’s PagedAttention), and increasingly, dedicated “prefill” and “decode” worker pools — each optimized for its distinct computational profile.

The scale of today’s token economy

The token economy has reached a scale that strains conventional intuition. Consider the disclosed figures from leading providers.

OpenAI has publicly stated its API platform processes approximately 6 billion tokens per minute — equivalent to roughly 8.64 trillion tokens per day if that rate held constant. Google has disclosed that its AI models process over 10 billion “direct API” tokens per minute, equating to approximately 14.4 trillion tokens per day for that segment alone, not accounting for internal workloads. On the consumer side, ChatGPT was reported as receiving approximately 2.5 billion prompts per day — a figure distinct from tokens, since each prompt generates input and output token streams that can vary widely by task type and context length.

Data point

No single audited public number exists for total global tokens per day across all providers. Estimates blend public and private traffic and may double-count workloads routed through cloud partners. Provider disclosures should be treated as directional, not as precise sector aggregates.

Some third-party syntheses adopt a rounded global baseline for ecosystem-scale planning — on the order of ~50 trillion tokens per day (~1.5 quadrillion per month) in discussions of agent-heavy traffic ( Fireworks AI). That number is not the same object as any single provider’s API throughput (and may overlap or double-count with them); it is mainly an alternate demand lens for back-of-envelope capacity exercises, as developed further below.

API pricing reflects this scale. Providers including Anthropic, Cohere, Google (Gemini Developer API), and OpenAI publish per-token rates that vary by model tier, input versus output, and whether cached context is applied — with output pricing sometimes explicitly including “thinking tokens” generated during extended reasoning. Subscription pricing provides a useful economic sanity check: OpenAI’s ChatGPT tiers range from $8 per month (Go) to $20 (Plus) to $200 (Pro), establishing a practical ceiling on what each subscriber can cost the platform unless strict usage limits or advertising revenues offset heavy users.

Mapping tokens to GPUs and electricity

Strategy requires converting token volumes into hardware realities. The following analysis provides a rigorous, assumption-transparent framework for doing so.

A throughput anchor: the 70B model on H100

MLPerf Inference v5.1 benchmark results offer the most credible public throughput anchor. One submission using NVIDIA H100 SXM systems — a 32-GPU configuration across four nodes — reported approximately 124,879 tokens per second in the Offline scenario for Llama 2 70B. Divided across 32 GPUs, this yields approximately 3,900 tokens per second per H100 in an offline-batch setting. The Server scenario for the same configuration showed around 122,274 tokens per second total, or roughly 3,820 tokens per second per H100.

A critical interpretive note: these are fleet-throughput figures under benchmark conditions. In production chat environments — where latency targets constrain how long a user waits for each output token — operators typically limit batch sizes to meet response-time service levels, reducing per-GPU throughput considerably in exchange for user experience quality. The benchmark numbers represent an engineering ceiling, not a real-world operating point.

Why batching changes everything

Batching is perhaps the single most powerful lever available to operators seeking to drive down cost per token. When multiple user requests are processed simultaneously, the model’s weights — the dominant memory consumer — are loaded once and shared across all requests in the batch, amortizing that fixed cost. The throughput gains are not incremental; they are multiplicative.

Static batching, however, is poorly suited to real-world traffic. Requests arrive at irregular intervals, outputs vary widely in length, and waiting for the slowest sequence in a batch to finish wastes GPU cycles. Continuous batching — where new requests slot in as others complete — keeps GPUs saturated without sacrificing latency for faster requests. Alongside this, KV cache management has emerged as a central operational challenge: each active sequence requires cache memory that grows proportionally with its context length, and cache fragmentation at scale becomes a meaningful source of waste. vLLM’s PagedAttention architecture addresses this directly and has become a reference implementation in production serving.

Empirical systems work bears this out: a 2025 comparative study of vLLM and Hugging Face Text Generation Inference reports materially higher GPU utilization for vLLM under high-concurrency workloads (with trade-offs for tail latency in low-concurrency settings), underscoring that stack choice and concurrency shape the utilization numbers that sit behind any tokens-per-GPU claim. Kolluru (2025), vLLM vs. TGI performance study.

Key formula

GPUs required ≈ (tokens per day) ÷ (tokens per second per GPU × 86,400 × utilization rate)

Worked examples: translating disclosed token volumes into GPU estimates

Assuming a planning figure of 3,900 tokens per second per H100 in an offline-like setting, one GPU can handle approximately 337 million tokens per day (3,900 × 86,400). This is an upper-bound planning figure; real systems operate below this level due to latency constraints, overhead, and demand spikes requiring headroom.

Applying this to OpenAI’s disclosed API volume: 8.64 trillion tokens per day divided by 337 million tokens per GPU per day yields approximately 25,600 H100-equivalent GPUs at 100% utilization. At a more realistic 50% utilization — accounting for traffic variability, latency headroom, and maintenance — the implied fleet requirement doubles to approximately 51,000 H100-equivalent GPUs. This is a unit conversion, not a fleet inventory claim. Its value is illustrative: it demonstrates why hyperscale token delivery rapidly translates into “tens of thousands of top-tier GPUs.”

For Google’s disclosed direct-API volume of 14.4 trillion tokens per day: at 100% utilization, approximately 42,700 H100-equivalent GPUs; at 50% utilization, approximately 85,000. The same caveats apply.

Cross-check: conservative throughput and a global baseline

The ~3,900 tokens per second figure is an offline-style MLPerf ceiling for a 70B-class model. A deliberately lower planning anchor appears in public inference-engine material — for example, Together AI reports on the order of ~117 tokens per second on Llama-2-70B-Chat, sometimes rounded to ~100 tokens per second per H100-class GPU in normalized models ( Together AI). At ~100 tokens per second sustained, one GPU delivers about 8.64 million tokens per day (100 × 86,400) — roughly 39× below the 337 million tokens per day upper bound from the MLPerf anchor. Fleet estimates move in direct proportion: the same token demand implies roughly 39× more GPUs when you swap in this conservative anchor instead of the benchmark one.

Pairing the ~50 trillion tokens per day ecosystem baseline (above) with ~100 tokens per second per GPU and 100% utilization yields on the order of ~5.8 million H100-equivalent GPUs — a different conversion than the OpenAI-only or Google API slices, because both total demand and per-GPU throughput changed. The energy reality

Token economics cannot be separated from energy economics. The physical layer of AI infrastructure is increasingly a binding constraint on the industry’s growth trajectory.

Power envelopes at the chip and rack level

At the inference-optimized end of the market, NVIDIA’s L4 GPU carries a maximum TDP of 72 watts — a figure relevant for “cheap token” scenarios where workloads migrate toward lower-power inference cards or custom silicon. At the frontier end, NVIDIA’s DGX GB200 rack-scale system consumes approximately 120 kilowatts per rack, with HPE’s NVL72 rack offering quoted at 132 kilowatts. Planning notes sometimes cite ~32 GPUs per rack (MLPerf-class cluster footprints) versus ~64 GPUs in dense liquid-cooled integration designs — both change per-rack load and cabling step-functions. This stands in stark contrast to conventional data center norms: industry surveys still find most facilities operating racks at 10 to 30 kilowatts, with very few exceeding that range. The shift to AI-dense infrastructure is not merely a hardware upgrade — it is a facilities redesign problem.

Estimating energy per million tokens

Energy per token can be estimated with a transparent three-variable model:

Energy per token ≈ (IT power × PUE) ÷ throughput

Applying this to the H100 benchmark anchor with defensible assumptions: 700 watts of average GPU load under production conditions (consistent with public discussion of H100 power profiles); a Power Usage Effectiveness (PUE) of 1.3 as a “good hyperscale” overhead approximation; and 3,900 tokens per second throughput for a 70B model in an offline-style batch setting.

Public sources do not give a single fixed PUE for AI data centers. The International Energy Agency notes that in modern data centers, servers often account for a large share of electricity demand, with storage and networking comparatively small, while cooling can range from roughly 7% of demand at efficient hyperscale sites to more than 30% at less efficient enterprise facilities — implying a wide spread in overhead factors across sites. That variability is why PUE should be treated as an explicit modeling choice, not a universal constant. See the IEA’s Energy demand from AI discussion for the underlying facility picture.

For PUE bracketing specifically: Google has reported fleet-wide figures near 1.09, while whole-industry averages are often discussed around 1.5–1.6 ( Google data center FAQ). The 1.3 value used in the worked example below sits between best-in-class hyperscale and typical enterprise overhead — a modeling choice, not a universal constant. (The 2040 stress-check tables later in this article use PUE 1.2 for that model’s simplicity; results shift modestly with PUE.)

Under these conditions, 1 million tokens requires approximately 256 seconds of GPU time. Energy consumed: 0.7 kW × 1.3 × (256 ÷ 3,600) ≈ 0.065 kilowatt-hours per million tokens.

That figure is highly regime-dependent. If latency constraints reduce effective throughput to 1,600 tokens per second per GPU — a plausible production figure for low-latency chat — the same arithmetic yields approximately 0.16 kilowatt-hours per million tokens. The ratio between these two scenarios is 2.5x, driven entirely by utilization assumptions. Any published “energy per token” figure that does not disclose its serving regime, model size, and batch level is analytically incomplete.

External benchmark

Epoch AI estimated a typical ChatGPT query using GPT-4o at roughly 0.3 watt-hours per query — a figure they also positioned as lower than some older, widely repeated estimates — yet still far above what peak-batch throughput mathematics alone would imply, because real serving carries overhead, latency targets, and lower effective batch sizes.

Global energy context and the Jevons paradox

The International Energy Agency estimates global electricity consumption from data centers at approximately 415 terawatt-hours in 2024, representing about 1.5% of global electricity use. Its base-case projection sees this roughly doubling to around 945 terawatt-hours by 2030 — with substantial uncertainty in either direction.

The strategic implication is a variant of Jevons’ Paradox: even as tokens become individually cheaper to produce, total token volume may grow faster than per-token efficiency gains, resulting in higher absolute energy demand. Efficiency and growth are not substitutes for one another at this scale. Boards overseeing energy-intensive operations — and policymakers setting grid investment priorities — should treat this dynamic as structural, not cyclical.

Infrastructure at scale: supply, demand, and bottlenecks

Synergy Research Group reported 1,189 large hyperscale data centers globally at the end of Q1 2025 — the most useful proxy for sites likely to host significant AI workloads, though not every hyperscale facility is GPU-dense. AI workloads are not neatly segregated; they run across hyperscale campuses, large colocation sites, and select enterprise environments.

Three forces driving token demand

Demand for tokens is being amplified along three distinct vectors simultaneously.

First, user growth. OpenAI reported very large weekly active user counts through 2025, accompanied by the 2.5-billion-prompts-per-day figure noted above. Each incremental user arriving on a platform adds to the baseline token load.

Second, context expansion. Larger context windows — and the growing prevalence of tool use, retrieval-augmented generation, and multi-modal inputs — mechanically increase the number of tokens processed per task. An interaction that once consumed 500 tokens may now consume 8,000 as systems are redesigned to leverage extended context (see e.g. ChatGPT release notes for how providers document ongoing context and capability changes).

Third, agentic architectures. Agent-style systems — which decompose user objectives into multi-step action plans, tool calls, and self-correction loops — can generate orders of magnitude more internal tokens per visible user action than simple chat exchanges. As agent deployments scale, their token footprint scales non-linearly with user count.

Supply-side efficiency: real, but bounded

On the supply side, meaningful cost reduction is occurring and can be expected to continue — within limits. Epoch AI reports that prices for equivalent AI performance fell sharply between 2023 and 2025, with variation across task types. Hardware is advancing rapidly: NVIDIA positions its GB200 NVL72 rack as delivering substantially higher performance per watt than H100-class air-cooled systems, with marketing claims on the order of 25× lower cost and energy than Hopper for some generative-AI workloads ( NVIDIA), and separately ~30× faster real-time LLM inference for GB200 NVL72 in some materials ( GB200 NVL72 product overview) — all vendor-framed, workload-dependent figures. Those same racks draw approximately 120 kilowatts, redefining facility requirements.

Software-layer efficiency gains compound hardware improvements. Continuous batching and advanced scheduling push utilization rates higher. PagedAttention-style cache management reduces memory waste. Quantization — moving models from FP16 to FP8 or INT4 precision — can yield significant reductions in GPU memory requirements and corresponding throughput improvements, with documented caveats around accuracy degradation and kernel complexity at lower bit widths. Oracle Cloud has published experiments describing large reductions in GPU requirements for certain serving configurations under INT4-style quantization.

Scenario planning note

Both “tokens per task” (driven by agentic behavior and context growth) and “tokens per second per GPU” (driven by hardware and software) can grow rapidly and simultaneously. Because of this compounding uncertainty, scenario planning consistently outperforms point forecasts in this domain.

Demand vs. supply (planning frame to 2035–2040)

A transparent way to bracket long-run token economics is to write demand and supply in the same units, then stress-test the drivers on both sides: 

Total tokens per year ≈ users × tasks per day × tokens per task × 365

Total supply (tokens per year) ≈ GPU count × (tokens per second per GPU) × (seconds per year) × utilization

The complication is that tokens per task can rise with context length and agentic behavior at the same time as tokens per second per GPU rises with better silicon and serving software — so both sides of the ledger can move quickly together. That is why the scenario-planning note above matters more than any single point forecast.

Where the system breaks: physical constraints

The binding constraints on AI infrastructure growth are increasingly physical rather than financial or algorithmic. Capital is already chasing gigawatt-class commitments: OpenAI has described Stargate-scale ambitions on the order of ~10 GW and, with Oracle, multigigawatt U.S. expansion ( OpenAI / Stargate announcements) — a different league than traditional incremental campus builds. Reuters Breakingviews has highlighted that even with declared AI capital expenditure commitments in the hundreds of billions of dollars, grid connection delays, shortages of skilled construction labor, and limited availability of specialized electrical equipment — transformers and turbines chief among them — create queues that money alone cannot resolve. Bloomberg reporting has documented parts shortages and construction delays in US data center builds, reinforcing that the critical path runs through physical supply chains, not balance sheets.

Geopolitical factors add a further layer of constraint. Reuters reporting has documented export control enforcement concerns, including restricted GPUs reaching sensitive buyers through circuitous procurement paths. NVIDIA’s own financial disclosures note licensing requirements affecting exports to China, outcomes that demonstrably affect business planning. The combination of physical scarcity and geopolitical friction means that the AI infrastructure buildout faces headwinds that cannot be engineered away.

The economics of serving one billion users

As AI platforms approach consumer scale, the cost structure bifurcates cleanly into variable and fixed components. Variable costs — dominated by GPU compute, power, and serving infrastructure overhead — scale with token volume. Fixed and semi-fixed costs — model training, data acquisition, safety research, and long-term data center construction — are amortized across the user base.

A useful managerial construct is “break-even tokens per subscription period.” If a plan is priced at $20 per month and the provider’s true all-in cost is X dollars per million tokens, a subscriber consumes approximately (20 ÷ X) million tokens before the economics invert. No provider has publicly disclosed audited cost-per-million-token figures, but the bracket can be established: retail API prices set an upper bound on what users pay; hardware and power mathematics reveal that at high utilization, electricity is frequently not the dominant cost component for frontier GPU workloads — it is the capital cost of the GPU itself, amortized across its service life.

This is precisely why “unlimited” plans universally come with fair-use provisions. Tokens carry a physical cost at every utilization level. When effective utilization falls — as it does for free-tier users arriving at irregular intervals — energy cost per token rises, even if the GPU is already depreciated. The economics of consumer AI pricing are therefore not fundamentally different from the economics of any other capacity-constrained service business.

“Unlimited” is a pricing fiction. Tokens have a physical cost at every utilization level.

Three scenarios for the next decade

Given the range of uncertainty across demand growth, hardware improvement rates, energy constraints, and geopolitical dynamics, scenario planning is more analytically honest than point forecasts. Three scenarios bracket the plausible distribution.

Best case: the token commodity era

Chip and software performance continue to improve rapidly, and infrastructure buildouts keep pace with demand through aggressive private investment and policy support for grid expansion. Token costs fall faster than demand grows, pushing the economics of common AI tasks toward pennies per day per user. Governments and educational institutions begin treating basic AI assistance as digital public infrastructure — analogous to broadband access — and expand access accordingly. In this scenario, AI’s economic value migrates upstream toward distribution, data, and product design rather than raw compute.

Worst case: infrastructure stagnation

Power and grid constraints stall buildout timelines. Export controls and supply-chain friction keep frontier GPUs scarce, limiting the pace of capacity expansion. Token costs stop falling — or rise during peak demand periods as capacity runs tight. A small number of firms with superior infrastructure position become gatekeepers, extracting rent from smaller providers who lack their own compute. Access to AI capability stratifies along infrastructure ownership lines, with significant implications for competitive dynamics across industries.

Most likely case: a tiered and fragmented market

The most probable outcome is a structurally tiered market. Standardized, frequently cached tasks — summarization, basic code completion, short-form content generation — become genuinely cheap, served by smaller quantized models at high utilization. Frontier tasks — extended reasoning, long-context analysis, multi-step agentic workflows — remain expensive, commanding premium pricing. Most consumer products will bifurcate into fast-and-cheap and slow-and-costly tiers, with advertising, metered limits, or tiered subscriptions managing the economics of free access. Providers with the best serving infrastructure and the most sophisticated caching strategies will maintain structural cost advantages.

Strategic implications: where value accumulates

If tokens become a commodity for standard tasks — as seems likely for the lower tiers of the market — economic value will shift to the layers above and below raw compute. Understanding where it accumulates is essential for strategy.

  • Distribution and workflow ownership. Whoever controls the interface layer — the enterprise software, the consumer application, the professional tool — captures value regardless of which model underlies it.
  • Data rights and proprietary data access. High-trust, high-specificity data — medical records, legal archives, financial history, proprietary enterprise knowledge — becomes increasingly valuable as a model input that commodity providers cannot replicate.
  • Product design and problem framing. As token costs fall, wasted tokens become the operational inefficiency to optimize. Products that frame problems precisely, reduce iteration loops, and return accurate outputs on fewer tokens will have structurally lower unit economics than competitors with poor prompt engineering.
  • Systems integration. Connecting models to existing enterprise tools, workflow systems, compliance infrastructure, and business rules is a durable source of switching costs that pure model capability cannot easily displace.
  • Trust, auditability, and regulatory compliance. In healthcare, legal, financial services, and government contexts, the premium is increasingly on verifiable outputs with audit trails — not on raw token cost. Providers who can demonstrate reliability and compliance will command margins independent of commodity token economics.

The strategic conclusion for builders and investors is direct: the model is not the product. It is a component. Durable competitive positions are built on workflows that the model serves, proprietary data that the model cannot access elsewhere, or distribution that the model relies on to reach users. The token economy will commoditize inference; it will not commoditize judgment, trust, or relationships.

A note on analytical honesty

Executives and strategists engaging with token economics should be alert to several sources of analytical noise in public discourse.

Provider token-volume disclosures may not be directly comparable. “Tokens per minute” figures can reflect different product surfaces, different tokenization schemes, or different counting conventions. They are informative as order-of-magnitude signals, not as audited sector aggregates.

Benchmark throughput figures — including the 3,900 tokens per second per H100 anchor used in this analysis — are derived under specific test conditions. Real chat-serving systems routinely operate at lower effective throughput to meet latency targets, manage demand spikes, and maintain memory headroom. Translating benchmark numbers into fleet estimates requires explicit utilization assumptions, and those assumptions dominate the result.

Energy-per-token estimates are extraordinarily sensitive to utilization, batching behavior, and infrastructure overhead. The same GPU hardware can appear highly efficient at maximum batch utilization and highly inefficient at low concurrency. Claims about AI energy consumption that do not disclose their modeling assumptions should be treated with appropriate skepticism.

Efficiency gains from precision reduction — moving from FP16 toward FP8 and 4-bit formats — can be large, but each step can run into accuracy floors, kernel complexity, and integration cost; it is plausible that the easy wins are front-loaded and that the pace of per-token efficiency improvement slows over time.

Demand trajectories are uncertain in both directions: agents and long context can raise tokens per task faster than many baselines assume, while cost, usage limits, or regulation could temper growth. Good analysis holds both tails in view instead of anchoring on a single narrative.

Finally, global energy consumption figures for AI workloads specifically — as distinct from data center electricity use broadly — remain uncertain even to major forecasting agencies, which publish scenario ranges rather than point estimates. The IEA’s projection of roughly doubling data center electricity consumption by 2030 is a base case, not a deterministic forecast.

The honest framing for any projection in this space is: directionally robust, parametrically uncertain. The direction of travel — more tokens, more GPUs, more electricity, falling per-token cost, rising total cost — is well-supported. The specific magnitudes depend on assumptions that should be stated explicitly and revisited as evidence accumulates.
